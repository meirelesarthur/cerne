GB CERNE

UI Web Guide

Padrões de interface para o painel GB CERNE. Toda geração de código deve seguir estas regras.

v2.0 · 27 mai 2026 · 13 seções
01

As Quatro Leis

Invioláveis. Aplicam-se a toda geração de código, revisão de PR e modificação de componente. Em caso de conflito entre qualquer outra seção deste guia, as Leis prevalecem.

Lei 1

Component-First — Componentização Obrigatória

Todo elemento visível na tela é um componente de src/components/ui/.

Proibido usar diretamente em páginas/telas:

  • <button>  <input>  <select>  <table>  <thead>  <tr>  <td>  <h1>–<h6>
  • Ao criar uma nova tela, apenas importar e chamar componentes do catálogo
  • Se o componente necessário não existe, criá-lo em src/components/ui/ antes de usá-lo
Lei 2

Fonte Única de Verdade — Propagação Global

Alterações em src/components/ui/ refletem em todas as telas automaticamente — esse é o objetivo.

Proibido em páginas/telas:

  • Sobrescrever estilos com style={} inline sobre um componente existente
  • Duplicar lógica de estilo que o componente já oferece via prop
  • Clonar/reimplementar um componente UI dentro de uma página

Extensões de comportamento são feitas adicionando props ao componente, nunca por patch local.

Lei 3

Fonte & Tokenização Acima de Tudo

Tipografia: única fonte permitida é Outfit. Qualquer outra é violação.

Tokens: todo valor de design vem de src/design/tokens.ts via t.*.

✓ Correto
color: t.color.brand[600] fontSize: t.font.size.sm padding: t.space[4] borderRadius: t.radius.DEFAULT
✕ Violação — hardcoded
color: '#059669' fontSize: '0.875rem' padding: '16px' borderRadius: '8px'
Lei 4

Commits Obrigatórios, Push sob Demanda

Após toda mudança concluída, um commit deve ser criado imediatamente — sem aguardar solicitação.

  • Push nunca é automático — somente quando o usuário solicitar explicitamente
  • Mensagens seguem Conventional Commits: feat: fix: style: refactor: docs:
  • Um commit por unidade lógica — não acumular alterações não relacionadas

Fonte

Outfit é a única fonte do projeto. Pesos disponíveis: 400, 500, 600, 700, 800. Qualquer outra fonte é violação da Lei 3.

Regras de Conteúdo

  • Números tabulares: colunas numéricas usam font-variant-numeric: tabular-nums
  • Reticências: caractere … (U+2026), nunca três pontos ...
  • Aspas curvas: "texto" em conteúdo editorial — nunca aspas retas
  • Non-breaking space: entre número e unidade — 10&nbsp;ha, R$&nbsp;100,00
  • Todos os estados desenhados: toda tela tem layout para vazio, esparso, denso e erro
05

Layout e Responsividade

Alinhamento

  • Usar CSS Grid e Flexbox — evitar dimensionamento via JavaScript
  • Alinhamento óptico: ajustar ±1–2 px quando percepção supera geometria (ícones ao lado de texto)
  • Alinhamento deliberado: cada elemento alinha com algo — grid, baseline, borda ou centro óptico

Breakpoints de Teste

ViewportLarguraVerificação
Mobile360 pxNenhum elemento vaza ou sobrepõe
Laptop1280 pxLayout padrão de desenvolvimento
Wide1920 pxLayouts não ficam excessivamente esticados

Scroll e Overflow

  • Corrigir overflow para evitar scrollbars indesejados
  • macOS: testar com Show Scroll Bars: Always em System Preferences
  • Modals e drawers: aplicar overscroll-behavior: contain

Links e Navegação

  • Links verdadeiros: usar <a> ou <Link> — nunca <button> para URLs
  • Deep-link tudo: filtros, aba, página atual e painéis expandidos persistem na URL
  • Back/Forward restaura estado e posição de scroll
06

Interações e Acessibilidade

Teclado e Foco

  • 100% operável por teclado — todos os fluxos críticos sem mouse
  • Usar :focus-visible em vez de :focus — não mostra ring em cliques de mouse
  • Focus ring padrão: t.glow.brand (box-shadow verde)
  • Modals e drawers prendem o foco enquanto abertos (focus trap)
✓ Correto
:focus-visible { box-shadow: t.glow.brand; }
✕ Ativa no click também
:focus { outline: 2px solid green; }

Hit Targets

ContextoTamanho mínimo
Desktop24 × 24 px
Mobile / touch44 × 44 px

Sem dead zones: se a área visual é interativa, toda ela é clicável.

Estados de Loading

  • Delay inicial antes de mostrar loading: 150–300 ms (evitar flash)
  • Tempo mínimo visível: 300–500 ms (evitar flicker)
  • Botões mantêm o rótulo original + spinner via prop loading

Atualizações Otimistas

  • UI responde imediatamente; reconcilia com resposta do servidor
  • Erro oferece Undo ou notificação clara com ação de correção

Ações Destrutivas

Sempre confirmação explícita (modal ou janela de Undo). Usar Button variant destructive — nunca estilizar exclusão como ação primária.

Semântica e ARIA

  • Semântica nativa antes de ARIA: preferir <button>, <a>, <label>, <table>
  • Estrutura de headings h1 → h2 → h3 nunca deve pular níveis
  • IconButton sempre com aria-label descritivo
  • Decoração: aria-hidden="true"
  • Toasts e validação inline: aria-live="polite"

Mobile

  • Inputs com font-size ≥ 16 px — evitar auto-zoom no iOS Safari
  • touch-action: manipulation em controles — elimina delay de double-tap
  • Nunca desabilitar zoom nem bloquear paste
07

Formulários

Submissão

  • Enter submete com um único campo focado; no último campo em formulários multi-campo
  • <textarea>: Ctrl/⌘ + Enter submete; Enter insere nova linha
  • Submit habilitado até o início da submissão; desabilitar durante o request com spinner
  • Nunca pré-desabilitar submit — permitir submissão incompleta para exibir feedback

Labels e Validação

  • Todo campo com <label> associado — FormField e FormSelect já implementam
  • Erros exibidos próximos ao campo, nunca apenas no topo
  • Ao submeter com erros: focar o primeiro campo com erro
  • Mensagens guiam a correção: não "campo inválido" — mas "CPF deve ter 11 dígitos"
  • spellcheck="false" em e-mails, códigos, usernames, tokens

Preenchimento Automático

  • Definir autocomplete e name significativos para habilitar autofill
  • Campos não-auth: autocomplete="off" para não disparar gerenciadores de senha
  • Fazer trim() antes de validar — evitar erros confusos por espaços
08

Animações e Transições

Princípios

  • Animar apenas quando clarifica causa/efeito ou adiciona delícia deliberada
  • Respeitar prefers-reduced-motion — fornecer variante reduzida
  • Animações são canceláveis por input do usuário
  • Nunca autoplay — animar em resposta a ações
/* prefers-reduced-motion obrigatório */
@media (prefers-reduced-motion: reduce) {
  * {
    animation-duration: 0.01ms !important;
    transition-duration: 0.01ms !important;
  }
}

Propriedades Seguras (GPU)

✓ GPU accelerated
transform: translateY(-4px) opacity: 0 filter: blur(4px)
✕ Causa reflow
width / height top / left margin / padding

Durações

TokenValorUso
t.animation.duration.fast150 msHover, foco
t.animation.duration.normal200 msMostrar/ocultar pequenos elementos
t.animation.duration.slow300 msModals, drawers
t.animation.duration.slower400 msPage transitions

Easing

  • t.animation.easing.easeOut — elementos que entram na tela
  • t.animation.easing.easeIn — elementos que saem da tela
  • t.animation.easing.standard — mudanças de tamanho
  • t.animation.easing.spring — drag, hover com "mola"

Transform Origin

  • Dropdown que abre abaixo → transformOrigin: 'top center'
  • Modal do centro → transformOrigin: 'center center'
  • Drawer lateral → transformOrigin: 'right center'
09

Estados da UI

Todo componente e toda tela deve ter todos os estados desenhados antes de ir para produção.

EstadoImplementação
LoadingSkeleton (listas/cards) ou prop loading do Button
VazioEmptyState com mensagem e ação de recuperação
ErroMensagem orientada à solução + ação de retry
SucessoFeedback positivo via toast ou Badge success
Esparso (1–3 itens)Layout bonito — não quebrado
Denso (+500 itens)Paginação + virtualização

Skeletons

  • Devem refletir a estrutura real do conteúdo final — evitar placeholders genéricos que causam layout shift
  • Usar variant text, rect ou circle conforme o tipo
  • Delay de ~150 ms antes de exibir — operações rápidas não devem piscar

Sem Dead Ends

Toda tela oferece um próximo passo ou caminho de recuperação:

  • Lista vazia → "Cadastrar primeiro item"
  • Erro de rede → "Tentar novamente"
  • Sem permissão → link para solicitar acesso

ProgressBar

No topo de cards/seções com operação em andamento: idle (invisível) → loading (brand indeterminado) → success → error.

Regras

  • Definir color-scheme: dark no <html> quando GBMode ativo — alinha scrollbars e controles nativos
11

Performance

Renderização

  • Monitorar re-renders com React DevTools — minimizar re-renders desnecessários
  • useMemo e useCallback apenas onde profiling confirmar necessidade
  • Listas com +100 itens visíveis: considerar virtualização

Rede

OperaçãoTarget
POST / PATCH / DELETE< 500 ms
Inputs de buscaDebounce de ~300 ms
Indicador de loadingExibir após 150–300 ms sem resposta

Assets

  • Dimensões explícitas em imagens — evitar CLS (Cumulative Layout Shift)
  • Lazy-load abaixo do fold
  • Fontes críticas com preload para evitar FOUT

Main Thread

Operações pesadas (parsing de grandes datasets, cálculos de relatório) fora do main thread. Web Workers para processamentos > 50 ms.

Copy e Mensagens de Interface

Voz e Tom

PrincípioCorretoEvitar
Voz ativa"Cadastre uma fazenda""Uma fazenda deve ser cadastrada"
Segunda pessoa"Sua safra foi salva""A safra do usuário foi salva"
Orientado à ação"Adicionar produtor""É necessário adicionar um produtor"

Labels e Botões

  • Específicos: "Salvar Safra" em vez de "Continuar" ou "OK"
  • Ações com consequência futura terminam com reticências: "Excluir safra…"
  • Estados de processamento: "Salvando…", "Carregando…"
  • Loading mantém o texto + spinner — nunca substituir por "Aguarde"

Erros

⚠️

Erros sempre guiam a saída. Não apenas "Erro ao salvar" — mas "Erro ao salvar: verifique sua conexão e tente novamente."

  • Linguagem positiva por padrão: problem-solving mesmo em estados de erro
  • Evitar linguagem técnica: "Timeout" → "A operação demorou mais que o esperado"

Números e Formatação

  • Numerais para contagens: "8 safras" em vez de "oito safras"
  • Moeda: 0 ou 2 decimais — nunca misturar no mesmo contexto
  • Datas e horas: locale-aware (pt-BR como padrão)

13

Governança e Commits

Conventional Commits

feat(modulo): fix(componente): style(ui): refactor(hook): docs(guidelines): chore(deps):

Formato: tipo(escopo): descrição imperativa em minúsculas

Checklist de PR

  • Todos os valores de design usam t.* — sem hardcode
  • Todos os componentes vêm de src/components/ui/
  • Suporte a ambos os temas (light e GBMode)
  • Estados da UI implementados (loading, vazio, erro)
  • Inputs com aria-label ou <label> associado
  • Botões apenas-ícone com aria-label
  • Animações com prefers-reduced-motion respeitado
  • Nenhum transition: all no código
  • Fonte Outfit em toda tipografia
  • Commit criado com mensagem Conventional Commits
