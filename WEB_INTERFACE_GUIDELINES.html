As Quatro Leis do Projeto

As Leis são invioláveis. Aplicam-se a toda geração de código, revisão de PR e modificação de componente.

Lei 1

Component-First — Componentização Obrigatória

Todo elemento visível na tela é um componente de src/components/ui/.

Proibido usar diretamente em páginas/telas:

  • <button>  <input>  <select>  <table>  <thead>  <tr>  <td>  <h1>–<h6>
  • Ao criar uma nova tela, apenas importar e chamar componentes do catálogo
  • Se o componente necessário não existe, criá-lo em src/components/ui/ antes de usá-lo na tela
  • Catálogo atual → ver Seção 3
Lei 2

Fonte Única de Verdade — Propagação Global

Alterações em src/components/ui/ refletem em todas as telas automaticamente — esse é o objetivo.

Proibido em páginas/telas:

  • Sobrescrever estilos com style={} inline sobre um componente existente
  • Duplicar lógica de estilo que o componente já oferece via prop
  • Clonar/reimplementar um componente UI dentro de uma página

Extensões de comportamento são feitas adicionando props ao componente, nunca por patch local.

Lei 3

Fonte & Tokenização Acima de Tudo

Tipografia: única fonte permitida é Outfit. Qualquer outra é violação.

Tokens: todo valor de design vem de src/design/tokens.ts via t.*.

✓ Correto
color: t.color.brand[600] fontSize: t.font.size.sm padding: t.space[4] borderRadius: t.radius.default
✕ Violação — hardcoded
color: '#059669' fontSize: '0.875rem' padding: '16px' borderRadius: '8px'
Lei 4

Commits Obrigatórios, Push sob Demanda

Após toda mudança concluída, um commit deve ser criado imediatamente — sem aguardar solicitação.

  • Push nunca é automático — somente quando o usuário solicitar explicitamente
  • Mensagens seguem Conventional Commits: feat: fix: style: refactor: docs:
  • Um commit por unidade lógica — não acumular alterações não relacionadas

Regra de hue consistency: em superfícies com background de cor (ex: card brand, GBMode), bordas, sombras e texto de suporte devem ser tingidos ao mesmo hue — nunca usar cinza neutro puro sobre fundo colorido.

Grid base de 4 px: t.space[1] = 4 px → t.space[20] = 80 px. Não usar valores fora da grade sem justificativa explícita.

Transições

Usar t.transition.* para consistência. Nunca usar transition: all — listar propriedades explicitamente.

✕ Evitar

Criando Novos Componentes

Tipografia e Conteúdo

Fonte

Outfit é a única fonte do projeto. Pesos disponíveis: 400, 500, 600, 700, 800. Qualquer outra fonte é violação da Lei 3.

// src/design/tokens.ts
fontFamily: t.font.family  // 'Outfit, sans-serif'

Hierarquia de Tamanhos

TokenpxUso
t.font.size.xs11Metadados, rodapés, labels de eixo em charts
t.font.size.sm12Labels de campos, badges
t.font.size.base13Texto de tabela, listas
t.font.size.md14Corpo padrão de formulários
t.font.size.lg15Títulos de cards, seções
t.font.size.xl16Títulos de página secundários
t.font.size.2xl22PageHeader titles
t.font.size.3xl26KPIs, números de destaque

Regras de Conteúdo

  • Números tabulares: em tabelas e comparações, usar font-variant-numeric: tabular-nums
  • Reticências tipográficas: usar o caractere … (U+2026), nunca três pontos ...
  • Aspas curvas: "texto" em conteúdo editorial — nunca aspas retas
  • Espaço non-breaking: 10&nbsp;kg, R$&nbsp;10,00, 5&nbsp;ha — evitar quebra entre número e unidade
  • Todos os estados desenhados: toda tela deve ter layout para vazio, esparso, denso e com erro

Layout e Responsividade

Grid e Alinhamento

  • Usar CSS Grid e Flexbox nativos — evitar dimensionamento via JavaScript
  • Alinhamento óptico: ajustar ±1–2 px quando percepção supera geometria (ex: ícones ao lado de texto)
  • Alinhamento deliberado: cada elemento deve alinhar com algo — grid, baseline, borda ou centro óptico

Breakpoints de Teste

ViewportLarguraObjetivo
Mobile360 pxVerificar que nenhum elemento vaza ou sobrepõe
Laptop1280 pxLayout padrão de desenvolvimento
Wide1920 pxVerificar que layouts não ficam excessivamente esticados

Barras de Scroll

  • Corrigir overflow para evitar scrollbars indesejados
  • Em macOS: testar com "Show Scroll Bars: Always" em System Preferences
  • Em modals e drawers: aplicar overscroll-behavior: contain

Links e Navegação

  • Links verdadeiros: usar <a> ou <Link> — nunca <button> ou <div onClick> para URLs
  • Deep-link tudo: filtros ativos, aba selecionada, página atual, painéis expandidos persistem na URL
  • Back/Forward do browser deve restaurar estado, incluindo posição de scroll

Interações e Acessibilidade

Teclado e Foco

  • 100% operável por teclado — todos os fluxos críticos sem mouse
  • Usar :focus-visible em vez de :focus — não mostra ring em cliques via mouse
  • Focus ring padrão: t.glow.brand (box-shadow verde)
  • Focus traps: modals e drawers prendem o foco enquanto abertos
✓ Correto
:focus-visible { box-shadow: t.glow.brand; }
✕ Evitar — ativa no click
:focus { outline: 2px solid green; }

Alvos de Clique (Hit Targets)

ContextoTamanho mínimo
Desktop24 × 24 px por elemento interativo
Mobile / touch44 × 44 px

Sem dead zones: se a área visual é interativa, toda ela deve ser clicável.

Estados de Loading

  • Delay inicial antes de mostrar loading: ~150–300 ms (evitar flash para operações rápidas)
  • Tempo mínimo visível do indicador: ~300–500 ms (evitar flicker)
  • Botões em loading mantêm o rótulo original + indicador — usar prop loading do Button

Atualizações Otimistas

  • UI responde imediatamente à ação do usuário
  • Reconciliar com resposta do servidor ao receber
  • Em caso de erro: mostrar opção de Undo ou notificação clara

Ações Destrutivas

Sempre exigir confirmação explícita (modal ou janela de Undo). Usar Button variant destructive — nunca estilizar exclusão como ação primária.

Acessibilidade Semântica

  • Semântica antes de ARIA: preferir <button>, <a>, <label>, <table> nativos
  • Headings hierárquicos: estrutura h1→h2→h3 não deve pular níveis
  • IconButton sempre tem aria-label descritivo
  • Elementos decorativos recebem aria-hidden="true"
  • Atualizações assíncronas (toasts) anunciadas com aria-live="polite"

Mobile

  • Inputs com font-size ≥ 16 px — evitar auto-zoom no iOS Safari
  • Usar touch-action: manipulation em controles — elimina delay de double-tap
  • Nunca desabilitar zoom do browser nem bloquear paste
7

Formulários

Comportamento de Submissão

  • Enter submete quando há um único campo focado
  • Em formulários multi-campo: Enter no último campo submete
  • <textarea>: Ctrl/⌘ + Enter submete; Enter insere nova linha
  • Submit fica habilitado até o início da submissão; desabilitar durante o request + spinner
  • Nunca pré-desabilitar o submit — permitir submissão incompleta para surfar feedback

Labels e Campos

  • Todo campo deve ter <label> associado — FormField e FormSelect já implementam
  • Erros exibidos próximos ao campo, nunca apenas no topo do formulário
  • Ao submeter com erros: focar o primeiro campo com erro

Validação

  • Não bloquear digitação — mostrar feedback após blur ou submit
  • Mensagens de erro guiam a correção: não "campo inválido", mas "CPF deve ter 11 dígitos"
  • spellcheck="false" em: e-mails, códigos, usernames, tokens de API

Tipos de Input Corretos

DadotypeinputMode
Numérico (não calculado)textnumeric
Telefonetel—
E-mailemail—
Decimaltextdecimal

Preenchimento Automático

  • Sempre definir autocomplete e name significativos para habilitar autofill
  • Não disparar gerenciadores de senha em campos não-auth: usar autocomplete="off"
  • Fazer trim() no valor antes de validar — evitar erros confusos por espaço

<select> no Windows

Definir explicitamente backgroundColor e color no FormSelect para evitar inconsistências visuais no Windows.

Animações e Transições

Princípios

  • Animar apenas quando clarifica causa/efeito ou adiciona delícia deliberada — nunca decorativo sem propósito
  • Respeitar prefers-reduced-motion — fornecer variante reduzida
/* Respeito a prefers-reduced-motion */
@media (prefers-reduced-motion: reduce) {
  * {
    animation-duration: 0.01ms !important;
    transition-duration: 0.01ms !important;
  }
}

Propriedades Seguras (GPU-accelerated)

Animar apenas estas para evitar reflow/repaint:

✓ GPU Accelerated
transform: translateY(-4px) opacity: 0 filter: blur(4px)
✕ Causa Reflow
width height top / left margin / padding

Durações

TokenValorUso
t.animation.duration.fast150 msHover, foco
t.animation.duration.normal200 msMostrar/ocultar pequenos elementos
t.animation.duration.slow300 msModals, drawers
t.animation.duration.slower400 msPage transitions

Easing

  • t.animation.easing.standard — mudanças de tamanho
  • t.animation.easing.easeOut — elementos que entram na tela
  • t.animation.easing.easeIn — elementos que saem da tela
  • t.animation.easing.spring — interações de drag, hover com "mola"

Transform Origin

  • Dropdown que abre abaixo: transformOrigin: 'top center'
  • Modal que cresce do centro: transformOrigin: 'center center'
  • Drawer lateral: transformOrigin: 'right center'

Regras Adicionais

  • Animações devem ser canceláveis por input do usuário
  • Nunca usar transition: all — listar propriedades explicitamente
  • Evitar autoplay — animar apenas em resposta a ações
  • SVG transforms: aplicar em wrappers <g> + transform-box: fill-box; transform-origin: center

Estados da UI

Todo componente e toda tela deve ter todos os estados desenhados antes de ir para produção.

Estados Obrigatórios

EstadoImplementação
LoadingSkeleton (listas/cards) ou prop loading do Button
VazioEmptyState com mensagem + ação de recuperação
ErroMensagem orientada à solução + ação de retry
SucessoFeedback positivo (toast ou Badge success)
Esparso (1–3 itens)Layout bonito, não quebrado
Denso (+500 itens)Paginação + virtualização

Skeletons

  • Devem refletir a estrutura real do conteúdo final — evitar placeholders genéricos que causam layout shift
  • Usar Skeleton variant text, rect ou circle conforme o tipo
  • Delay de ~150 ms antes de exibir — operações rápidas não devem piscar

Sem Dead Ends

Toda tela deve oferecer um próximo passo ou caminho de recuperação:

  • Lista vazia → botão "Cadastrar primeiro item"
  • Erro de rede → botão "Tentar novamente"
  • Sem permissão → link para solicitar acesso

ProgressBar

Usar ProgressBar no topo de cards/seções com operação em andamento: idle (invisível) → loading (indeterminado brand) → success → error.

Temas — Light e GBMode

Gerenciado por src/context/ThemeContext.tsx. Acessado via useTheme().

PropriedadeLightGBMode

Brand principal#059669#10b981
ToggleuseTheme().toggle()
Aplicação no DOMdata-theme="light"data-theme="gb"

Regras de Tema

  • Todo componente em src/components/ui/ deve suportar ambos os temas
  • Usar useTheme().colors para superfícies e isGbMode para valores específicos
  • Nunca hardcodar cores que dependem do tema
  • Em GBMode: bordas, sombras e textos de suporte devem usar hues verdes (não cinza neutro)
  • Definir color-scheme: dark no <html> quando GBMode ativo — alinha scrollbars e controles nativos do OS

Performance

Renderização

  • Monitorar re-renders com React DevTools — minimizar re-renders desnecessários
  • useMemo e useCallback apenas onde profiling confirmar necessidade
  • Listas com +100 itens visíveis: considerar virtualização

Latência de Rede

OperaçãoTarget
POST / PATCH / DELETEResposta < 500 ms
Inputs de buscaDebounce de ~300 ms antes do request
Indicador de loadingExibir após 150–300 ms sem resposta

Imagens e Assets

  • Sempre definir width e height explícitos em imagens — evitar CLS
  • Lazy-load imagens abaixo do fold
  • Fontes críticas com preload para evitar FOUT

Main Thread

Operações pesadas (parsing de grandes datasets, cálculos de relatório) fora do main thread. Considerar Web Workers para processamentos > 50 ms.

Copy e Mensagens de Interface

Voz e Tom

PrincípioCorretoEvitar
Voz ativa"Cadastre uma fazenda""Uma fazenda deve ser cadastrada"
Segunda pessoa"Sua safra foi salva""A safra do usuário foi salva"
Orientado à ação"Adicionar produtor""É necessário adicionar um produtor"

Labels e Botões

  • Labels claros e específicos: "Salvar Safra" em vez de "Continuar" ou "OK"
  • Ações com consequência futura terminam com reticências: "Excluir safra…", "Arquivar fazenda…"
  • Ações em processamento: "Salvando…", "Carregando…"
  • Botões em loading mantêm o texto + spinner — não substituir por "Aguarde"

Mensagens de Erro

⚠️

Erros sempre guiam a saída. Não apenas "Erro ao salvar" — mas "Erro ao salvar: verifique sua conexão e tente novamente."

  • Linguagem positiva por padrão: problem-solving mesmo em estados de erro
  • Evitar linguagem técnica: "Timeout" → "A operação demorou mais que o esperado"

Números e Formatação

  • Numerais para contagens: "8 safras" em vez de "oito safras"
  • Moeda: 0 ou 2 decimais — nunca misturar no mesmo contexto
  • Datas e horas: formato locale-aware (pt-BR como padrão)
  • Colunas numéricas de tabelas: font-variant-numeric: tabular-nums

Terminologia Consistente

UsarEvitar
FazendaPropriedade, Campo, Terra
SafraCiclo, Temporada
ProdutorAgricultor (salvo contexto específico)
CadastrarInserir, Adicionar (escolher um e manter)
ExcluirDeletar, Remover (em contextos destrutivos)
13

Governança e Commits

Conventional Commits

feat(modulo): fix(componente): style(ui): refactor(hook): docs(guidelines): chore(deps):

Formato: tipo(escopo): descrição imperativa em minúsculas

Checklist de PR

  • Todos os valores de design usam t.* — sem valores hardcoded
  • Todos os componentes vêm de src/components/ui/
  • Suporte a ambos os temas (light e GBMode)
  • Todos os estados da UI implementados (loading, vazio, erro)
  • Inputs com aria-label ou <label> associado
  • Botões apenas-ícone com aria-label descritivo
  • Animações com prefers-reduced-motion respeitado
  • Nenhum transition: all no código
  • Fonte Outfit em toda tipografia
  • Commit criado com mensagem Conventional Commits

Atualizando Este Documento

Este documento deve ser atualizado sempre que:

  • Um novo componente é adicionado ao catálogo (src/components/ui/)
  • Um novo token é criado em tokens.ts
  • Uma nova Lei ou regra é aprovada pela equipe
  • Um padrão de interação é definido para casos recorrentes
